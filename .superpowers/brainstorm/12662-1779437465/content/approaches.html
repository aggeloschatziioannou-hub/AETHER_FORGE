Architecture Approaches

Three ways to structure the system

A — Direct SQLite (Simplest)

Canvas Panel → Next.js API Route → better-sqlite3 → .db file

No separate backend. Next.js API routes read/write SQLite directly. MCP proxy runs Python subprocesses for non-SQLite DBs later.

Pros

  • Zero extra processes
  • Fastest to build
  • Same deployment as app

Cons

  • SQLite-only unless extended
  • Python not used

B — MCP Proxy (Recommended)

Canvas Panel → Next.js API Route → FastAPI Proxy → MCP Servers → .db files
                                    → Direct SQLite → .db files

Python FastAPI backend sits between the React frontend and databases. Panels never connect directly to DBs — they go through the proxy. Future panels reuse the same proxy for their own data needs.

Pros

  • True to your vision C (any DB)
  • Python stack you wanted
  • Reusable for future panels
  • Panels stay stateless/light

Cons

  • Extra process to manage
  • More code to write upfront

C — Full Monolith (FastAPI + embedded Next.js build)

FastAPI serves both API + static Next.js build.
Single process. Single port. Single deploy.

Next.js builds to static files, FastAPI serves them alongside the API. One process to start, one port to remember.

Pros

  • Simplest deployment
  • Same Python stack everywhere

Cons

  • Lose Next.js SSR/API routes
  • Next.js is better at frontend
  • Harder to develop (two build systems)
